## 0.2 各角色研讨意见

### 产品经理意见

> "竞品分析不能盲目做，得先定好分析框架。"

1. **关于产品边界**: "个人助手"范围极广（日程管理、智能对话、生活服务、文件处理...），如果不圈定核心场景，竞品对标就会漫无目的。建议先明确3个以内的核心场景。
2. **关于竞品对标方法**: 第一部分的竞品分析已经很全面，但建议进一步按层级分类对标：
   - **直接竞品**：功能定位相似的App（如豆包、Kimi）
   - **间接竞品**：系统级助手（小爱、小艺、Siri）
   - **潜在竞品**：可能扩展到同赛道的产品（钉钉AI、飞书AI）
3. **关于差异化**: 第1.5节的五个切入方向都是好方向，但在公司规范约束下，不是所有方向都可行。需要结合公司政策缩小范围。

### 架构师意见

> "技术选型决定了后面80%的工作量。"

1. **关于技术选型前置条件**: 第二部分的分析很扎实，但有一个前提：**公司是否有技术栈强约束？** 如果公司指定Java而非Kotlin，或指定某个特定框架，那当前推荐方案需要全面调整。
2. **关于鸿蒙的优先级判断**: 鸿蒙适配是最大的技术不确定性。建议在Phase 1阶段同步做一个鸿蒙的最小技术验证（比如Hello World级别的ArkTS应用），避免Phase 3启动时才发现阻塞性问题。
3. **关于AI模型过渡策略（重要更新）**: 已明确公司有自研模型但仍在开发中，前期需用业界开源模型过渡。这带来几个关键架构要求：
   - **模型可插拔架构是刚需**：不是"Nice to have"，而是"Must have"。架构必须支持低成本切换模型，未来从开源切到自研不需要大改。
   - **需要定义统一的模型接口协议**：前期对接开源模型时就要按这个协议来，自研模型成熟后按同一协议接入。
   - **Prompt/上下文管理要模型无关**：不同模型的prompt格式、上下文窗口大小、能力边界不同，需要抽象层屏蔽差异。
   - **前期开源模型选择建议从"私有部署"优先考虑**：既然公司有自研意愿，说明对数据和模型有自主可控的诉求。开源模型私有部署（如DeepSeek开源版、Qwen开源版）可能比调用第三方API更符合公司方向。
   - **需要和自研模型团队对齐接口规范**：越早对齐，未来切换成本越低。

### 合规顾问意见

> "公司规范是硬约束，必须前置确认，否则后面返工成本极高。"

1. **关于合规前置**: 第三部分列出了需要确认的公司规范，但这些不应该是"注意事项"，而应该是**阻塞性前置条件**。建议将其提升为P0优先级。
2. **需要特别关注的合规风险**:
   - **AI相关法规**: 《生成式人工智能服务管理暂行办法》要求备案、内容审核、标识义务——这对独立App影响重大
   - **各应用商店审核差异**: Google Play / App Store / 华为应用市场各有不同的AI应用审核要求
   - **数据主权**: 如果使用海外AI API（OpenAI/Claude），用户数据可能涉及跨境传输
3. **关于公司规范收集**: 建议立即启动以下规范文档的收集（见0.3节的行动清单）

### 项目经理意见

> "节奏和资源必须前置规划，不能只有技术路线没有项目路线。"

1. **关于里程碑**: 第2.4节的分阶段路线（Phase 1→2→3）是合理的，但缺少每个阶段的验证目标。建议明确：
   - Phase 1（Android MVP）验证什么？技术可行性？还是用户需求？还是两者？
   - 每个Phase的"完成"标准是什么？
2. **关于资源**: 当前文档没有涉及团队规模、技能需求。一个人做和五个人做，方案会截然不同。
3. **关于风险管理**: 需要建立风险清单并持续跟踪（见0.4节）

### UX设计师意见

> "助手类App的交互范式选择，会影响整个技术架构。"

1. **关于设计策略的前置决策**:
   - 统一设计语言 vs 遵循各平台规范（Material Design / HIG / HarmonyOS Design）？这个决策会反向影响跨平台策略的选择。
   - 助手类App的核心交互范式：对话式？卡片式？混合式？需要在开发之前确定。
2. **关于公司设计规范**: 是否有公司统一的设计系统/组件库？品牌色彩、Logo使用规范？这些需要前置确认。
3. **关于竞品体验对标**: 建议补充对竞品的交互体验维度分析，不只看功能列表。

### 安全工程师意见

> "安全架构需要在设计阶段就介入，而不是开发完了再做安全审查。"

1. **关于威胁建模**: 个人助手App涉及大量用户隐私数据（对话内容、联系人、日历等），需要在架构设计阶段完成威胁建模（STRIDE模型）。
2. **关于API安全**: AI API Key的管理方式需要前置设计——Key放在客户端极易被逆向提取，建议通过自建后端代理。
3. **关于端侧安全**: 如果采用端侧推理，模型文件的完整性验证、防篡改机制需要考虑。

# 第一部分：竞品分析

## 1.1 市场全景

全球智能个人助手市场 2025 年规模约 **165.8 亿美元**，预计 2035 年达 **2682.6 亿美元**（CAGR 32.1%）。中国 AI 大模型市场 2026 年预计突破 **7000 亿元**。

当前市场分为两大阵营：

```
┌─────────────────────────────────────────────────────────┐
│                    个人助手市场格局                        │
├────────────────────────┬────────────────────────────────┤

├────────────────────────┼────────────────────────────────┤
│ Siri (Apple)           │ ChatGPT (OpenAI) — 7.69亿MAU  │

```

## 1.2 国际主流竞品对比

### Apple Siri / Apple Intelligence
- **AI能力**: 端侧Foundation Models + Private Cloud Compute；iOS 26将引入上下文感知Siri
- **优势**: 深度系统集成、隐私保护行业标杆、8420万用户、全设备生态
- **劣势**: AI能力落后竞品、上下文Siri多次延期、中文弱、Apple Intelligence未在中国大陆上线

### Google Gemini (原Assistant)
- **AI能力**: Gemini 2.5 Pro、Deep Research、多模态
- **优势**: 8880万用户、搜索引擎优势、Google生态整合
- **劣势**: Assistant→Gemini过渡混乱延期至2026年3月、隐私问题、中国不可用

### Microsoft Copilot
- **AI能力**: GPT-5、Tasks自动化工作流、最多32人协作
- **优势**: 办公场景最强、企业级集成、任务自动化领先
- **劣势**: 个人场景偏弱、订阅贵($30/月)、中国市场受限

### ChatGPT
- **AI能力**: GPT-5/o3、Agent模式（自动执行研究/做PPT/订票）、Tasks定时任务
- **优势**: 全球MAU最高(7.69亿)、品牌最强、Agent能力行业领先
- **劣势**: 费用高($20-200/月)、中国需VPN、隐私争议

## 1.4 行业五大趋势

### 趋势一：大模型驱动的对话式AI
传统语音助手全面向LLM升级。从"命令式交互"到"对话式协作"的范式转换。

### 趋势二：多模态交互成为标配
视觉理解、语音升级、图像/视频生成、手势与空间感知。预计2027年成为行业标准。

### 趋势三：端侧AI与隐私优先
Apple端侧Foundation Models、Gemma 3n离线运行、llama.cpp手机端推理。**隐私正从"可选项"升级为"差异化竞争力"**。

### 趋势四：Agent能力（从对话到行动）
ChatGPT Agent自主执行任务、Copilot Tasks跨App自动化、豆包向系统级Agent演进。AI Agent市场CAGR 46.3%。

### 趋势五：社交化与个性化
腾讯元宝派AI社交、Copilot Groups协作、AI具备"个性"和"外观"。

## 1.5 市场机会与"二狗"的切入点

### 可切入的差异化方向

| 机会 | 描述 | 竞争强度 | 适合独立开发者 |
|------|------|---------|--------------|
| **隐私优先的本地化AI助手** | 数据不出设备，端侧推理为主 | 低 | 高 |
| **轻量化"数字管家"** | 极简、快速、低资源占用的AI助手 | 中 | 高 |
| **垂直场景深耕** | 聚焦一个核心场景做到极致 | 中 | 高 |
| **跨平台统一助手** | 打破生态围墙、跨品牌管理 | 高 | 中 |
| **情感+生产力混合型** | 有温度的交互 + 真正完成任务 | 中 | 中 |

### 竞品成败关键教训

**成功要素**:
1. 清晰的产品策略 > 技术堆叠
2. 功能强大且易用的平衡
3. 生态集成深度（没有生态的AI助手难建护城河）
4. "越用越好用"的个性化正循环
5. 极致性价比

**失败教训**:
1. Kimi：过早商业化 + 缺乏生态 → 月活跌出前五
2. 天猫精灵：频繁更名 + 定位混乱 → 品牌认知崩塌
3. 传统语音助手：固守命令式交互 → 被视为"只会设闹钟的工具"
4. 功能堆砌而非场景解决 → **"少而精"往往比"大而全"更有效**

## 2.1 跨平台框架对比

### 关键结论
- **uni-app/Taro 不适合**: 个人助手需要深度系统集成（后台服务、通知、电话、无障碍），Web-based框架在这些场景短板明显
- **Flutter鸿蒙版最有潜力**: 三端UI一致性好，已有京东等大厂验证，但仍是Beta
- **KMP适合共享业务逻辑**: 但UI层需各端独立开发，鸿蒙无官方支持

- 自研模型为主力，开源模型仅作降级/兜底

### AI调用架构设计

```kotlin
// KMP 共享层 - AI Service 抽象
interface LLMService {
    suspend fun chat(messages: List<Message>, config: ModelConfig): Flow<String>
    suspend fun functionCall(messages: List<Message>, tools: List<Tool>): ToolCallResult
}

// 可切换的实现（模型可插拔 — 为自研模型过渡而设计）
class DeepSeekService : LLMService      // 前期主力（开源过渡）
class QwenService : LLMService          // 备选
class LocalLLMService : LLMService      // 端侧 llama.cpp
class OpenAIService : LLMService        // 用户自定义
class InHouseModelService : LLMService  // 公司自研模型（预留接口，待模型就绪后实现）

// 智能路由（支持过渡期灰度切换）
class LLMRouter(
    networkMonitor: NetworkMonitor,
    privacyConfig: PrivacyConfig,
    modelConfig: ModelConfig,          // 模型切换配置（可远程下发）
) {
    fun selectService(task: TaskType): LLMService = when {
        !networkMonitor.isConnected -> LocalLLMService()      // 离线→端侧
        task.isPrivacySensitive -> LocalLLMService()           // 隐私→端侧
        modelConfig.useInHouseModel(task) -> InHouseModelService() // 自研模型灰度
        else -> DeepSeekService()                              // 默认→开源云端
    }
}
```

## 2.5 记忆系统设计

> **这是"个人助手"区别于"聊天工具"的核心能力。** 没有记忆的助手每次对话都是陌生人，不可能做到"越用越懂你"。

### 设计哲学：像人一样记忆，而不是像数据库

> **核心理念**: 二狗的记忆不是一个完美的数据库，而是像人一样——有选择地记、需要时间回忆、可能记得不完全准确、会形成自己的抽象理解和判断。
>
> 完美的记忆反而让AI显得冰冷。真实的、有"人味"的记忆，才能让用户觉得二狗是个有温度的伙伴。

#### 五个核心特征

| 特征 | 人类类比 | 二狗的表现 | 举例 |
|------|---------|-----------|------|
| **选择性记忆** | 人不会记住所有事，重要的、有情感的、反复出现的才记得住 | 二狗对重要信息、反复提及的事、有情感标记的内容记忆更深 | 你提过三次"想学吉他" → 记忆强化；随口说的"今天吃了面条" → 可能不记 |
| **回忆需要时间** | 人回想事情需要"想一下"，不是瞬间检索 | 回忆非即时信息时，可以有短暂的"让我想想..."过程 | "你之前跟我说过一个餐厅...让我想想...是不是上个月说的那家川菜馆？" |
| **记忆不完美** | 人的记忆会模糊、混淆、遗忘细节 | 时间越久的记忆越模糊，可能记得大概但忘了细节 | "你好像上个月提过一次旅行计划，但我不太记得具体去哪了，你再说说？" |
| **抽象与判断** | 人会从经历中总结规律、形成偏好判断 | 二狗从多次对话中抽象出用户的性格、偏好、习惯模式 | 多次对话后总结出"你是个做事很有计划的人"，而不是只记录每个事实 |
| **短期鲜活** | 刚发生的事记得清清楚楚 | 最近的对话内容清晰准确，越早的越模糊 | 今天聊的细节一清二楚，上周的只记个大概 |

#### 与传统方案的对比

```
传统AI助手（数据库思维）          二狗（认知思维）
┌────────────────────┐         ┌────────────────────┐
│ 完美记录一切         │         │ 选择性记住重要的      │
│ 检索精准、即时返回    │         │ 回忆需要时间          │
│ 永远100%准确         │         │ 可能模糊、可能不准     │
│ 只是信息存取         │         │ 会抽象、会判断        │
│ 冰冷但高效          │         │ 有温度、像朋友        │
│                    │         │                    │
│ → 感觉像数据库      │         │ → 感觉像一个          │
│   "你3月1日说了X"    │         │   真正认识你的朋友     │
│                    │         │   "我记得你好像       │
│                    │         │    提过类似的事..."    │
└────────────────────┘         └────────────────────┘
```

#### 技术实现思路

| 人类特征 | 技术映射 | 实现方式 |
|---------|---------|---------|
| **选择性记忆** | 记忆重要度评分 | LLM评估每条信息的重要度(1-5)，低分的不入库或快速衰减 |
| **反复强化** | 重复计数 + 权重提升 | 同一话题被提及多次 → 记忆强度+1，权重提升 |
| **情感标记** | 情感分析标签 | 带有强情感的内容（开心/难过/重要）获得更高记忆权重 |
| **回忆延迟** | 检索时的展示策略 | 非即时记忆加入"让我想想"的自然过渡，不是瞬间甩出精确结果 |
| **记忆模糊** | 时间衰减 + 细节丢失 | 存储时保留完整信息，但召回时根据时间距离，LLM以"大概记得"的口吻回忆 |
| **抽象判断** | 定期总结 + 用户画像更新 | 每隔N次对话，LLM对近期对话做一次抽象总结，更新用户画像 |
| **遗忘** | 衰减+淘汰 | 长期未被召回且重要度低的记忆逐渐衰减，最终归档或删除 |

#### 记忆强度模型

```
记忆强度 = 基础重要度 × 重复强化系数 × 情感权重 × 时间衰减

其中：
- 基础重要度: LLM评估(1-5分)
- 重复强化: 被提及N次 → 系数 = 1 + log(N)
- 情感权重: 中性=1.0, 轻度情感=1.2, 强情感=1.5
- 时间衰减: e^(-λt)，λ根据记忆类型不同而不同
  - 事实类（生日、地址）: λ很小，几乎不衰减
  - 事件类（"上周吃了什么"）: λ较大，快速衰减
  - 偏好类（"喜欢吃辣"）: λ中等，多次确认后趋于稳定

记忆强度 > 阈值 → 可被召回
记忆强度 < 淘汰线 → 归档/删除
```

### 各层详细设计

#### 工作记忆（Working Memory）— 当前对话

| 要素 | 说明 |
|------|------|
| **是什么** | 当前会话的多轮对话历史，直接作为LLM的context传入 |
| **关键问题** | 上下文窗口有限（如DeepSeek 64K tokens），长对话需要截断或压缩 |
| **策略** | 滑动窗口：保留最近N轮完整对话 + 更早轮次的摘要 |
| **实现** | 内存中维护 `List<Message>`，超过阈值时触发摘要压缩 |

#### 短期记忆（Short-term Memory）— 近期上下文

| 要素 | 说明 |
|------|------|
| **是什么** | 跨会话的近期信息——上次聊到哪了、最近关注什么、待办事项 |
| **关键问题** | 需要在新会话开始时自动"回忆"相关近期上下文 |
| **策略** | 每次会话结束时，LLM自动生成会话摘要，存入本地数据库 |
| **实现** | SQLite存储会话摘要，新会话开始时注入最近几条摘要作为system prompt |
| **衰减** | 按时间衰减权重，7天以上的短期记忆降级或归档到长期记忆 |

#### 长期记忆（Long-term Memory）— 用户知识库

| 要素 | 说明 |
|------|------|
| **是什么** | 关于用户的持久化知识：偏好、事实、习惯、关系、重要经历 |
| **关键问题** | 如何从对话中自动提取有价值的信息？如何高效检索相关记忆？ |
| **提取策略** | 对话中检测到关键信息时（通过LLM判断），自动提取为结构化记忆 |
| **存储** | 结构化部分用SQLite（如用户资料），非结构化部分用向量数据库做语义检索 |
| **检索** | 新对话时，根据当前话题做语义相似度匹配，召回相关长期记忆注入prompt |
| **更新** | 新信息与旧记忆冲突时，更新而非追加（如"我换工作了"→更新职业信息） |

#### 多模态记忆（Multimodal Memory）— 照片与图像

> 用户需求：希望二狗能记住照片。照片是生活中重要的信息载体——拍个名片、拍个菜单、拍个药盒、存张截图，都是"记忆"。

| 要素 | 说明 |
|------|------|
| **是什么** | 用户主动保存或对话中发送的图片，以及从图片中提取的结构化信息 |
| **核心思路** | 照片本身是"原始记忆"，从照片中提取的文字/信息是"可检索的记忆" |
| **存储策略** | 图片文件存本地专用目录 + 图片元信息和提取的文字/标签存SQLite |
| **用户交互** | 拍照/选图 → 二狗理解图片内容 → 生成摘要/标签 → 存入记忆 → 后续可通过对话检索 |

**典型使用场景**:

| 场景 | 用户操作 | 二狗行为 |
|------|---------|---------|
| 拍名片 | "帮我记住这张名片" | 识别姓名、电话、公司、职位 → 存入联系人记忆 |
| 拍菜单 | "这家店不错，记一下" | 识别餐厅名、菜品 → 关联地理位置 → 存入美食记忆 |
| 存处方 | "这是我的处方" | 识别药品名、用法用量 → 存入健康记忆 → 可设用药提醒 |
| 截图保存 | "记住这个信息" | 提取截图中的关键文字 → 打标签存入 |
| 拍证件 | "存一下我的身份证" | 识别信息 → **高安全等级加密存储** |
| 旅行照片 | "这是我在三亚拍的" | 提取地点/时间/内容描述 → 关联旅行记忆 |

**技术实现**:

```
用户拍照/选图
     │
     ▼
┌──────────────┐
│ 图片理解       │  ← 多模态LLM（云端）或 OCR（端侧/云端）
│ - OCR文字识别  │
│ - 场景/物体识别 │
│ - 内容摘要生成  │
└──────┬───────┘
       │
       ▼
┌──────────────────────────────────────────┐
│ 结构化存储                                 │
│                                           │
│ 图片文件 → App私有目录/加密存储              │
│ 图片元信息 → SQLite                         │
│   - 原图路径                               │
│   - 缩略图路径                              │
│   - OCR提取的文字                           │
│   - LLM生成的内容摘要                       │
│   - 标签/分类（名片/票据/证件/风景/...）     │
│   - 时间戳、地理位置（如有）                  │
│   - 安全等级（普通/敏感/高敏感）              │
│   - 关联的文字记忆ID（与长期记忆关联）        │
└──────────────────────────────────────────┘
       │
       ▼
  后续对话中可检索：
  "我上次拍的那个名片上的电话号码是多少？"
  "我在三亚吃的那家餐厅叫什么？"
  → 通过标签/关键词/语义匹配找到对应图片记忆
```

**图片理解方案**:

| 方案 | 能力 | 适用场景 | 成本 |
|------|------|---------|------|
| **多模态LLM API** (DeepSeek-VL/通义千问VL) | 图片理解、场景描述、信息提取 | 通用场景，效果最好 | 按token计费 |
| **OCR服务** (科大讯飞/百度OCR) | 文字识别 | 名片、票据、文档类 | 免费额度+按量 |
| **端侧OCR** (ML Kit / PaddleOCR Lite) | 基础文字识别 | 离线场景、隐私敏感 | 免费 |

> **推荐策略**: 云端多模态LLM做"理解"（生成摘要和标签），端侧OCR做"兜底"（离线时至少能识别文字）。

**存储空间管理**:

| 策略 | 说明 |
|------|------|
| **缩略图** | 保存时自动生成缩略图，列表展示用缩略图，减少内存占用 |
| **压缩** | 原图按质量压缩（如80%质量JPEG），证件类保留原图 |
| **存储上限** | 设置总存储上限（如2GB），接近上限时提醒用户清理 |
| **云端备份（可选）** | 未来可支持加密后备份到用户的云存储（如WebDAV/坚果云），非必需 |

**隐私特别注意**:

| 要点 | 说明 |
|------|------|
| **证件类高安全存储** | 身份证、银行卡等照片必须加密存储，且提供独立的访问保护（如指纹/PIN） |
| **图片不上云（默认）** | 图片文件默认只存本地，发送到云端LLM做理解时，用完即删、不留存 |
| **EXIF信息** | 存储时保留地理位置等EXIF信息供检索，但分享时应自动剥离 |

### 端侧向量检索方案（长期记忆检索用）

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **SQLite + 余弦相似度** | Embedding存SQLite，查询时计算相似度 | 记忆条数<1万，简单可靠 |
| **Hnswlib (C++ 库)** | 高效近似最近邻搜索，可通过JNI调用 | 记忆条数较多，需要高性能 |
| **自建轻量方案** | 分类标签 + 关键词匹配 + 时间衰减 | MVP阶段，不依赖向量检索 |

> **MVP建议**: 先用 **SQLite + 关键词/标签匹配** 做简易版长期记忆，不上向量数据库。验证记忆功能的产品价值后，再升级为向量检索。
>
> **Next项目验证**: 在Next中已实际运行验证——SQLite + `ORDER BY last_accessed_at DESC LIMIT 20` + 记忆分类(fact/habit/personality/intent)，效果已满足日常使用。会话摘要（短期记忆）在实践中发现不是必需的，20条历史消息 + 长期记忆注入已够用。详见 [06-Next项目已验证经验.md](./06-Next项目已验证经验.md)。

### Embedding方案（将文本转为向量）

| 方案 | 说明 | 推荐 |
|------|------|------|
| **云端Embedding API** | DeepSeek/通义千问 Embedding接口，效果好 | 有网络时使用 |
| **端侧小模型** | all-MiniLM-L6 (ONNX Runtime)，约22MB | 离线兜底 |

### 记忆的隐私与安全

| 要点 | 说明 |
|------|------|
| **本地优先** | 所有记忆数据默认存储在设备本地，不上传云端 |
| **加密存储** | 记忆数据库使用 SQLCipher 或 Android Keystore 加密 |
| **用户可控** | 用户可以查看、编辑、删除任何一条记忆 |
| **明示同意** | 首次使用时明确告知用户记忆功能，允许关闭 |
| **遗忘权** | 提供"清除所有记忆"功能，真正删除而非软删除 |

### 竞品记忆能力对标

| 产品 | 记忆能力 | 说明 |
|------|---------|------|
| **ChatGPT** | Memory功能 | 自动从对话中提取记忆，用户可查看管理，跨会话生效 |
| **Claude** | 无持久记忆 | 仅支持项目级knowledge，无跨会话记忆 |
| **豆包** | 基础记忆 | 记住用户名、偏好等基本信息 |
| **Kimi** | 弱 | 上下文长但无持久化记忆 |

> **"二狗"的机会**: 大部分竞品的记忆功能还很初级。如果"二狗"能做到**真正"越用越懂你"**——记住你的习惯、喜好、重要日程、社交关系——这本身就是强差异化。

### 分阶段实施建议

| 阶段 | 记忆能力 | 实现方式 |
|------|---------|---------|
| **POC** | 工作记忆（多轮对话） | 内存维护对话历史，滑动窗口截断 |
| **MVP** | + 短期记忆（会话摘要）+ 基础长期记忆（用户资料/偏好） | SQLite存储摘要和结构化信息，关键词匹配检索 |
| **MVP** | + 基础图片记忆（拍照保存+OCR文字提取） | 图片存本地目录，OCR提取文字+手动打标签存SQLite |
| **V2** | + 完整长期记忆（语义检索）+ 智能图片理解 | 向量Embedding + 相似度检索 + 多模态LLM自动生成图片摘要和标签 |
| **V3** | + 主动记忆（"我记得你上周说..."）+ 图片关联回忆 | 时间感知 + 主动关联 + 情境触发回忆（"你上次拍的那个..."） |

# 第五部分：UX设计策略分析

> **设计师视角**: 助手类App的交互范式选择，会反向影响技术架构。设计策略需要在开发之前确定。

## 5.1 交互范式选择

| 范式 | 代表产品 | 优势 | 劣势 | 适合场景 |
|------|---------|------|------|---------|
| **纯对话式** | ChatGPT、DeepSeek | 学习成本低、最自然 | 信息密度低、回溯困难 | 通用AI对话 |
| **卡片式** | Google Assistant、小爱 | 结构化展示、操作明确 | 灵活度低、开发成本高 | 系统级助手、快捷操作 |
| **混合式** | 豆包、Copilot | 兼顾对话灵活性和结构化展示 | 设计复杂度高 | 功能丰富的个人助手 |
| **嵌入式/悬浮式** | Gemini Overlay、华为小艺 | 全局可达、不打断当前操作 | 占用屏幕空间、系统权限要求高 | 系统级集成 |

**初步建议**: "二狗"作为独立App，推荐**混合式**（对话为主线 + 功能卡片为辅助），兼顾灵活性和结构化。

## 5.2 各平台设计规范差异

| 维度 | Android (Material 3) | iOS (HIG) | HarmonyOS Design |
|------|---------------------|-----------|-------------------|
| 导航模式 | Bottom Navigation / Navigation Rail | Tab Bar / Navigation Stack | Bottom Tab / Side Navigation |
| 手势 | 返回手势、下拉刷新 | 边缘滑动返回 | 多级返回、胶囊按钮 |
| 弹窗 | Dialog / Bottom Sheet | Alert / Action Sheet | Dialog / Panel |
| 字体 | Roboto / 系统字体 | SF Pro / PingFang | HarmonyOS Sans |
| 圆角 | 大圆角 (28dp) | 中圆角 (13pt) | 中大圆角 |

**待决策**: 采用统一设计语言（跨平台一致）还是遵循各平台原生规范（用户更熟悉）？

> **建议**: 如果选择"Android原生先行"的技术策略，设计也应先以Material 3为主，后续平台适配时再考虑平台差异。

## 5.3 待前置定义的设计交付物

1. **设计系统/组件库** — 是否有公司统一的？如果没有，需要新建
2. **核心交互流程原型** — 至少覆盖：首次启动、对话交互、语音唤醒、系统集成操作
3. **信息架构** — App的功能模块组织方式
4. **无障碍/适老化** — 是否有合规要求？

> **架构师/项目经理/安全工程师共同视角**: 工程基础设施是项目的"地基"。在写第一行业务代码之前，这些能力应该先规划清楚，否则后面补的成本远高于一开始就搭好。

```
├──────────────────┼──────────────────┼───────────────────────────────┤

│ CI/CD 流水线      │ 崩溃收集          │ 管理员看板                     │
```

## 7.2 各项详细分析

### 一、版本控制与代码管理

| 项目 | 选项 | 建议 | 需确认 |
|------|------|------|--------|
| Git托管平台 | GitHub / GitLab / Gitee / 公司自建 | 取决于公司规范 | **公司是否有指定平台？** |
| 分支策略 | Git Flow / Trunk-Based / GitHub Flow | 小团队推荐 GitHub Flow（简单） | 公司是否有分支管理规范？ |
| 代码审查 | PR/MR Review | 建议至少1人Review后合入 | 是否有审查流程要求？ |
| 提交规范 | Conventional Commits | 统一commit message格式，便于自动生成changelog | — |

### 二、CI/CD 流水线

| 阶段 | 内容 | 工具选项 | 说明 |
|------|------|---------|------|
| 代码检查 | Lint、静态分析 | ktlint / detekt / Android Lint | 提交时自动检查 |
| 编译构建 | Debug/Release APK | Gradle + GitHub Actions / Jenkins / 公司流水线 | 每次PR触发 |
| 自动化测试 | 单元测试、UI测试 | JUnit5 + Espresso / Robolectric | 构建后自动运行 |
| 产物管理 | APK/AAB归档 | Maven仓库 / 蒲公英 / 公司内部分发 | 版本可追溯 |
| 部署发布 | 应用市场上架 | Fastlane / 手动 | 后期自动化 |

> **需确认**: 公司是否有统一的CI/CD平台？是否有构建环境要求？

### 三、日志系统

| 层级 | 内容 | 方案选项 | 优先级 |
|------|------|---------|--------|
| **客户端日志** | App运行日志、用户操作日志 | Timber(本地) + 上报服务 | P0 — MVP就需要 |
| **AI调用日志** | 模型请求/响应、耗时、Token用量 | 自建（记录到后端） | P0 — 成本监控必备 |
| **后端日志** | API请求、错误、性能 | ELK / Loki+Grafana / 阿里云SLS / 公司方案 | P1 — 有后端时需要 |
| **日志脱敏** | 用户对话内容等敏感信息不能明文入日志 | 脱敏规则 + 审计 | P0 — 安全要求 |

### 四、监控与告警

| 维度 | 监控什么 | 方案选项 | 优先级 |
|------|---------|---------|--------|
| **崩溃监控** | App Crash、ANR | Firebase Crashlytics / Bugly(腾讯) / Sentry | P0 — 第一天就要接 |
| **性能监控(APM)** | 启动耗时、页面加载、内存、卡顿 | Firebase Performance / 听云 / 自研埋点 | P1 |
| **AI服务监控** | 模型API可用性、响应延迟、错误率 | 自建仪表盘 + 告警 | P0 — 核心体验 |
| **业务监控** | 对话成功率、用户活跃、功能使用率 | 自建或接入数据平台 | P1 |
| **告警通知** | 异常自动通知相关人 | 钉钉/飞书/企微机器人 / PagerDuty | P1 |

> **国内注意**: Firebase在国内不稳定，如果面向国内用户，**Bugly（腾讯）或 Sentry 自部署**更可靠。

### 五、管理员看板 (Admin Dashboard)

| 功能模块 | 说明 | 优先级 |
|---------|------|--------|
| **用户管理** | 用户列表、封禁/解封、用户画像 | P1 |
| **AI用量统计** | Token消耗、API调用量、成本统计 | P0 — 控制成本 |
| **内容审核** | AI回复的合规审查、敏感内容告警 | P0 — 法规要求 |
| **模型管理** | 模型切换、灰度比例配置、A/B测试控制 | P1 — 模型过渡期需要 |
| **配置下发** | 远程配置（开关、参数、公告等） | P1 |
| **数据看板** | DAU/MAU、留存率、功能使用分布 | P2 |

> **架构建议**: Admin看板不需要第一天做完，但**数据埋点和API必须从第一天设计好**，否则后面没数据可看。

### 六、项目管理与协作

| 项目 | 选项 | 建议 | 需确认 |
|------|------|------|--------|
| 任务管理 | Jira / Linear / GitHub Projects / 飞书项目 / 禅道 | 取决于公司规范和团队习惯 | **公司是否有指定工具？** |
| 文档管理 | Confluence / 飞书文档 / Notion / 语雀 | 统一一个地方存文档 | 公司是否有指定平台？ |
| 设计协作 | Figma / 即时设计 / MasterGo | 设计稿交付和标注 | — |
| 沟通工具 | 飞书 / 钉钉 / 企微 / Slack | — | 公司已有 |

### 七、测试策略

```
测试金字塔:

        /  E2E测试  \          ← 少量，覆盖核心流程
       / UI组件测试   \        ← 关键页面
      / 集成测试        \      ← API调用、数据库、模型服务
     / 单元测试           \    ← 大量，覆盖业务逻辑
    ──────────────────────
```

| 测试类型 | 工具 | 覆盖范围 | 优先级 |
|---------|------|---------|--------|
| 单元测试 | JUnit5 + MockK | 业务逻辑、ViewModel、Repository | P0 |
| UI测试 | Compose Testing + Espresso | 关键交互流程 | P1 |
| 集成测试 | Ktor MockEngine / WireMock | AI API调用、数据存储 | P1 |
| E2E测试 | Maestro / UI Automator | 核心用户场景端到端 | P2 |

> **建议**: MVP阶段至少做到单元测试覆盖核心业务逻辑，CI里跑通。不追求覆盖率数字，但关键路径必须有测试。

### 八、安全与审计

| 项目 | 说明 | 优先级 |
|------|------|--------|
| API密钥管理 | AI API Key不能硬编码在客户端，通过后端代理或安全存储 | P0 |
| 数据加密 | 本地对话记录加密(Android Keystore)、传输HTTPS | P0 |
| 安全审计日志 | 关键操作（登录、数据导出、配置变更）记录审计日志 | P1 |
| 依赖安全扫描 | 第三方库漏洞扫描 | Dependabot / Snyk / OWASP Dependency-Check | P1 |
| 代码安全扫描 | 静态安全分析 | CodeQL / SonarQube | P1 |
| 混淆与加固 | Release包ProGuard/R8混淆 + 加固（梆梆/360） | P1 |
